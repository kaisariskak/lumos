# Дизайн: динамические показатели отчёта для группы

## Контекст

Сейчас экран `Отчёт` строится из фиксированного набора системных показателей (`IbadatCategory.all`), а админка поддерживает только отдельные "доп. показатели" через `CustomCategory`. В результате проект одновременно поддерживает две разные модели:

- фиксированные показатели с максимумами в `ibadat_group_settings`
- дополнительные показатели в `group_custom_categories`

Новая цель — полностью отказаться от фиксированного набора и перейти к одной динамической модели: каждая группа сама определяет, какие показатели существуют в её отчётах.

## Цель

Сделать так, чтобы:

- администратор создавал показатели отдельно для каждой группы
- вкладка `Отчёт` отображала только показатели текущей группы
- у каждого показателя были собственные иконка, цвет, единица измерения и максимальное значение
- введённое значение могло превышать максимум
- проценты выполнения рассчитывались по максимуму, но ограничивались `100%`

## Не-цели

- сохранение совместимости со старой схемой фиксированных показателей
- миграция старых данных
- глобальный справочник показателей между несколькими группами
- возврат к системным предустановленным показателям

Пользователь явно указал, что базу можно очистить для повторного тестирования, поэтому дизайн оптимизирован под чистую новую схему без обратной совместимости.

## Пользовательские сценарии

### 1. Администратор настраивает показатели группы

Администратор открывает вкладку `Админ`, выбирает группу и управляет списком её показателей. Для каждого показателя он задаёт:

- название
- иконку
- цвет
- единицу измерения
- максимальное значение

После сохранения показатель становится частью конфигурации группы.

### 2. Пользователь открывает вкладку `Отчёт`

Пользователь входит в приложение и уже привязан к группе. Во вкладке `Отчёт` он видит только те карточки показателей, которые администратор создал для этой группы. Фиксированные системные карточки больше не показываются.

### 3. Пользователь заполняет значение выше максимума

Если для показателя задан максимум `100`, пользователь может ввести `130`. Значение `130` сохраняется как фактическое значение. Для расчёта процента выполнения этот показатель считается выполненным на `100%`.

### 4. Администратор или пользователь смотрит сводный прогресс

Проценты по отдельным показателям и общий прогресс строятся на базе групповых максимумов. Каждый показатель даёт вклад не выше `100%`, даже если его фактическое значение больше максимума.

## Архитектурное решение

### Единая модель групповых показателей

Вместо разделения на "системные" и "дополнительные" показатели приложение использует одну сущность группового показателя.

Предлагаемая сущность:

- `id`
- `groupId`
- `name`
- `icon`
- `colorValue`
- `unit`
- `maxValue`
- `orderIndex`
- `createdAt`

Источник правды для списка показателей группы — отдельная таблица групповых показателей. Экран `Отчёт` и расчёты прогресса читают данные только из неё.

### Единая модель значений отчёта

Текущий `IbadatReport` хранит фиксированные поля вроде `quranPages`, `bookPages` и т. д. Это нужно заменить на модель, где сам отчёт хранит общую "шапку", а значения лежат отдельно по каждому показателю группы.

Новая структура:

- `report` — метаинформация отчёта (`id`, `userId`, `groupId`, `periodId`, `month`, `year`, `submittedAt`, `updatedAt`)
- `report_metric_value` — строки со значениями (`reportId`, `metricId`, `value`)

Такой подход позволяет строить отчёт из произвольного набора показателей без добавления новых колонок в базу и без жёсткой связи с `IbadatCategory`.

## Изменения в данных и репозиториях

### 1. Групповые показатели

Нужно заменить текущую модель `CustomCategory` на полноценную сущность группового показателя. По сути, существующий блок "доп. показатели" становится основным механизмом настройки всех показателей группы.

Ожидаемые изменения:

- новая или переименованная модель для группового показателя
- репозиторий с операциями:
  - получить показатели группы
  - создать показатель
  - удалить показатель
  - при необходимости позже обновить и переупорядочить показатель

### 2. Значения отчётов

Нужно отделить значения показателей от фиксированных полей `IbadatReport`.

Ожидаемые изменения:

- `IbadatReport` хранит только метаинформацию отчёта
- отдельная коллекция значений показателей хранится как `Map<String, int>` по `metricId`
- репозиторий отчётов загружает и сохраняет значения через отдельную таблицу значений

### 3. Настройки максимумов

`IbadatGroupSettings` больше не должен быть источником максимумов для показателей отчёта, потому что максимум теперь принадлежит самому групповому показателю.

Следствие:

- логика `ibadat_group_settings` перестаёт участвовать в расчётах отчёта
- экран админки больше не редактирует максимумы через фиксированный список категорий

## Изменения в UI

### Вкладка `Админ`

Блок управления показателями группы должен стать основным, а не дополнительным.

Новый ожидаемый UX:

- список всех показателей выбранной группы
- кнопка добавления показателя
- форма создания показателя:
  - название
  - иконка
  - цвет
  - единица измерения
  - максимальное значение
- удаление показателя

На первом этапе допустимо не делать редактирование существующего показателя и drag-and-drop сортировку. Достаточно добавления, удаления и порядка по времени создания или `orderIndex`.

### Вкладка `Отчёт`

Экран больше не должен итерироваться по `IbadatCategory.all`.

Вместо этого:

- загрузить список показателей текущей группы
- для каждого показателя построить карточку
- иконка и цвет берутся из данных показателя
- текст максимума показывается в карточке
- значение редактируется через текущие контролы

Ограничение по максимуму на ввод не применяется. Это влияет на детали UI:

- слайдер не должен быть единственным источником ввода, если он искусственно ограничивает значение максимумом
- кнопка `+` не должна блокироваться на максимуме
- быстрые кнопки `25/50/75/100%` остаются удобным способом заполнить целевые значения, но не ограничивают ручной ввод

Практически это означает, что текущий UI стоит перевести в режим, где счётчик `- / value / +` является основным способом ввода, а слайдер либо удаляется, либо используется только в диапазоне `0..maxValue` как быстрый инструмент для целевых значений. Для первого релиза приоритет — корректная логика данных, даже если визуальный ввод будет немного упрощён.

### Детальный экран и домашний экран

Экран деталей, карточки сводки и расчёты в `HomeScreen` сейчас завязаны на фиксированные ключи категорий и `group settings`. Их нужно перевести на динамический список показателей группы.

Это касается:

- отображения отчётных значений
- расчёта прогресса пользователя
- расчёта среднего прогресса группы
- любых summary-row, которые сейчас читают только фиксированные поля отчёта

## Расчёт процентов

### Процент по одному показателю

Если:

- `maxValue <= 0`, показатель не участвует в процентах, чтобы избежать деления на ноль
- `value <= 0`, вклад равен `0`
- иначе вклад = `min(value / maxValue, 1.0)`

Примеры:

- `value=25`, `max=100` → `25%`
- `value=100`, `max=100` → `100%`
- `value=140`, `max=100` → `100%`

### Общий процент отчёта

Общий процент пользователя = среднее арифметическое вкладов всех показателей группы, у которых задан валидный `maxValue > 0`.

Если у группы нет показателей, общий прогресс = `0`.

### Групповой прогресс

Групповой прогресс рассчитывается как среднее по пользовательским прогрессам, как и сейчас, но источник данных меняется с фиксированных категорий на динамические показатели группы.

## Ошибки и граничные случаи

### Группа без показателей

Если администратор ещё не создал показатели для группы:

- вкладка `Отчёт` показывает пустое состояние с понятным сообщением
- сохранение пустого отчёта не требуется

### Удаление показателя

Если показатель удалён после того, как по нему уже были значения:

- он перестаёт отображаться в новых отчётах и расчётах
- старые строки значений по удалённому `metricId` могут остаться в базе, но не должны ломать загрузку

На первом этапе достаточно того, чтобы UI и репозитории игнорировали осиротевшие значения и не падали на них.

### Значение больше максимума

Это валидный сценарий. UI не показывает это как ошибку. В прогрессе значение просто ограничивается `100%`.

## Тестирование

Нужно покрыть как минимум:

- создание группового показателя
- загрузку списка показателей группы
- сохранение значений отчёта по динамическим показателям
- повторное открытие отчёта с уже сохранёнными значениями
- расчёт процента при значении меньше максимума
- расчёт процента при значении больше максимума
- пустую группу без показателей
- игнорирование удалённого показателя при загрузке значений

Особенно важно протестировать, что после удаления зависимости от `IbadatCategory.all` экран `Отчёт`, `HomeScreen` и `DetailScreen` продолжают работать только на основе групповых показателей.

## Влияние на текущий код

Основные точки изменений:

- `lib/models/ibadat_category.dart` — выводится из активной логики отчётов
- `lib/models/custom_category.dart` — заменяется или расширяется до полноценной модели группового показателя
- `lib/models/ibadat_report.dart` — убираются фиксированные поля показателей
- `lib/models/ibadat_group_settings.dart` — перестаёт быть источником максимумов отчёта
- `lib/repositories/custom_category_repository.dart` — превращается в репозиторий групповых показателей
- `lib/repositories/ibadat_report_repository.dart` — сохраняет и читает только шапку отчёта + динамические значения
- `lib/screens/report/report_editor_screen.dart` — строится полностью динамически
- `lib/screens/admin/admin_screen.dart` — управляет полным списком показателей группы
- `lib/screens/home/home_screen.dart` — считает проценты по динамическим показателям
- `lib/screens/detail/detail_screen.dart` — показывает детали по динамическим показателям

## Решения по объёму первого этапа

Чтобы не раздувать первую реализацию, в первый этап входят только:

- динамический список показателей на группу
- создание показателя в админке
- удаление показателя
- выбор иконки
- выбор цвета
- выбор единицы измерения
- задание максимума
- динамический экран отчёта
- динамические расчёты прогресса

Во второй этап при необходимости можно вынести:

- редактирование уже созданного показателя
- ручную сортировку
- подтверждения при удалении с более детальной обработкой старых значений
- более продвинутый UI ввода значений выше максимума

## Итог

Система отчётов переходит от фиксированного набора системных категорий к динамической конфигурации показателей на уровне группы. Это убирает двойную модель данных, делает вкладку `Отчёт` полностью зависимой от настроек админа и напрямую соответствует пользовательскому сценарию: пользователь входит в группу и видит только те показатели, которые администратор создал именно для этой группы.
