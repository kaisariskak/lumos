# Дизайн PDF-документации по Flutter-проекту

## Цель

Подготовить подробную техническую PDF-документацию по Flutter-проекту `reportdeepen` с формальным разбором файлов, классов, функций, виджетов, компонентов и последовательности вызовов, ориентированную на начинающего разработчика.

## Формат результата

Результатом должен стать PDF-документ на русском языке.

Документ должен быть:

- формальным по структуре;
- подробным по содержанию;
- понятным для новичка по объяснениям;
- последовательным по описанию потока выполнения;
- пригодным как для чтения подряд, так и для поиска по разделам.

## Основной подход

Использовать гибридную структуру документации:

1. Сначала показать общую архитектуру приложения и поток запуска.
2. Затем разобрать проект по файлам и слоям.
3. После этого отдельно описать последовательности вызовов между объектами.
4. В конце добавить справочные таблицы и словарь терминов.

Этот подход выбран потому, что он одновременно даёт читателю общую картину системы и позволяет детально изучать конкретные классы и файлы.

## Границы документации

Документация должна охватывать:

- точку входа приложения;
- глобальную инициализацию;
- настройки темы и локализации;
- аутентификацию и вход в приложение;
- модели данных;
- репозитории доступа к данным;
- сервисы;
- экраны;
- повторно используемые виджеты;
- вспомогательные утилиты;
- тесты в той степени, в которой они помогают понять архитектуру.

Документация не обязана:

- переписывать весь исходный код дословно;
- документировать Flutter SDK целиком;
- объяснять каждый стандартный Flutter-виджет, если он используется типовым образом;
- превращаться в справочник по Dart/Flutter вне контекста текущего проекта.

## Структура PDF

### 1. Введение

- Назначение проекта.
- Общий стек технологий.
- Какую задачу решает приложение.

### 2. Архитектурная карта

- Общая схема слоёв проекта.
- Роль папок `main`, `authentication`, `screens`, `repositories`, `models`, `services`, `widgets`, `theme`, `l10n`, `utils`.

### 3. Последовательность запуска приложения

- Что происходит в `main.dart`.
- Как инициализируется Supabase.
- Как обрабатываются deep link и OAuth callback.
- Как запускается `IbadatApp`.

### 4. Пофайловый разбор

Для каждого файла должно быть описано:

- зачем файл существует;
- какие классы, функции или константы он содержит;
- какие зависимости использует;
- кто вызывает этот файл или его классы;
- какие объекты он создаёт;
- какие данные принимает и возвращает;
- как он влияет на поток приложения.

### 5. Последовательность вызовов

Нужно отдельно показать цепочки выполнения:

- запуск приложения;
- вход пользователя;
- прохождение `AuthGate`;
- работа PIN;
- ввод invite-кода;
- переход в `MainScaffold`;
- дальнейшая навигация по основным экранам.

### 6. Таблицы связей

Добавить таблицы:

- файл → его назначение;
- класс → где объявлен;
- экран → какие модели и репозитории использует;
- репозиторий → с какими таблицами/сущностями работает;
- сервис → какую проблему решает.

### 7. Словарь терминов

Объяснить для новичка термины:

- `Widget`;
- `StatefulWidget`;
- `State`;
- `Repository`;
- `Model`;
- `Service`;
- `Provider` в текущем контексте;
- `SupabaseClient`;
- `BuildContext`;
- `Navigator`;
- `MaterialApp`;
- `Scaffold`.

## Стиль объяснения

Документ должен использовать:

- простой и точный язык;
- технически корректные определения;
- пояснение причин, а не только перечисление фактов;
- последовательное описание от общего к частному;
- пояснения в духе “что это”, “зачем нужно”, “кто вызывает”, “что вызывает дальше”.

Документ не должен:

- предполагать, что читатель уже хорошо знает Flutter;
- перескакивать между слоями без объяснения связи;
- использовать жаргон без расшифровки;
- оставлять большие блоки кода без пояснений.

## Критерии качества

Документация считается успешной, если:

- новичок может проследить полный путь запуска приложения;
- по каждому важному файлу понятно, зачем он нужен;
- по каждому ключевому классу понятно, кто его создаёт и где он используется;
- читатель понимает, как данные проходят через модели, репозитории, сервисы и экраны;
- PDF можно использовать как базовую внутреннюю техническую документацию проекта.
