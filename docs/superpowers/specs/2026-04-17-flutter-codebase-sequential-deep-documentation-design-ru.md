# Дизайн последовательной глубокой документации по Flutter-проекту

## Цель

Подготовить один большой документ на русском языке, который последовательно и максимально подробно объясняет весь Flutter-код проекта `reportdeepen` для начинающего разработчика.

## Формат результата

Результат должен быть не в виде короткой технической справки, а в виде цельного большого документа, который читается подряд как пошаговое прохождение по коду.

Документ должен:

- идти в порядке выполнения приложения;
- подробно объяснять каждый файл;
- подробно объяснять каждый класс;
- подробно объяснять каждый метод;
- подробно объяснять каждый виджет;
- подробно объяснять строки и логические блоки кода;
- постоянно показывать, какой объект кого вызывает и что происходит дальше.

## Основной принцип

Документ должен следовать не просто структуре папок, а реальному жизненному циклу приложения:

1. запуск приложения;
2. создание корневого приложения;
3. прохождение стартовой логики авторизации;
4. переход по экранам;
5. работа с данными;
6. вспомогательные слои.

То есть читатель должен видеть не только "что лежит в файле", но и "когда этот файл включается в работу".

## Порядок документа

### 1. `main.dart`

Нужно объяснить:

- каждую строку;
- зачем вызывается `WidgetsFlutterBinding.ensureInitialized()`;
- зачем и как инициализируется Supabase;
- как работают deep links;
- почему в конце вызывается `runApp`.

### 2. `ibadat_app.dart`

Нужно объяснить:

- что такое `StatefulWidget` в контексте корневого приложения;
- зачем хранятся listeners;
- как работают `LocaleProvider` и `AccentProvider`;
- как строится `MaterialApp`.

### 3. `AuthGate`

Это центральный файл стартовой логики, поэтому он должен быть разобран особенно подробно:

- состояние;
- подписка на auth events;
- проверка текущей сессии;
- проверка PIN;
- загрузка профиля;
- переходы на разные экраны;
- сценарии ошибок;
- активация invite-кода.

### 4. Экранный слой в порядке пользовательского пути

Сначала должны идти экраны, которые пользователь реально видит раньше:

1. `IbadatAuthorization`
2. `PinScreen`
3. `InviteCodeScreen`
4. `MainScaffold`
5. дочерние экраны из `MainScaffold`

Для каждого экрана нужно объяснить:

- конструктор;
- параметры;
- состояние;
- жизненный цикл;
- обработчики событий;
- `build()` и структуру UI;
- переходы к другим экранам;
- какие репозитории и модели используются.

### 5. Модели

После экранов должны идти модели, чтобы читатель уже понимал, где именно они используются.

Для каждой модели нужно объяснить:

- назначение;
- смысл каждого поля;
- `fromJson`;
- `toJson`;
- вычисляемые свойства;
- `copyWith`, если есть;
- где эта модель применяется в приложении.

### 6. Репозитории

После моделей должны идти репозитории.

Для каждого репозитория нужно объяснить:

- с какой таблицей он работает;
- что делает каждый публичный метод;
- какие параметры принимает;
- какие данные возвращает;
- какой экран или сервис обычно вызывает этот метод;
- как метод влияет на дальнейший поток приложения.

### 7. Сервисы

Отдельно объяснить сервисы, особенно `PinService`, с детальным разбором каждой функции.

### 8. `theme`, `l10n`, `config`

Подробно объяснить, как работают:

- глобальный конфиг;
- тема;
- цветовой акцент;
- локализация;
- выбор языка.

### 9. `widgets` и `utils`

В конце объяснить:

- переиспользуемые виджеты;
- утилиты;
- зачем они вынесены отдельно;
- кто их использует.

## Степень детализации

Документ должен объяснять код на двух уровнях одновременно:

### Уровень 1: построчное объяснение

Если строка важна для понимания, нужно объяснить:

- что она делает;
- почему она здесь нужна;
- что было бы без неё.

### Уровень 2: объяснение логических блоков

Если несколько строк работают как единый механизм, нужно:

- сначала объяснить блок целиком;
- затем разъяснить роль ключевых строк внутри него.

Это позволит сохранить и глубину, и читаемость.

## Стиль объяснения

Документ должен быть:

- очень подробным;
- последовательным;
- простым по языку;
- формальным по структуре;
- ориентированным на новичка.

Нужно постоянно использовать пояснения в формате:

- что это;
- зачем это нужно;
- кто это вызывает;
- что это вызывает дальше;
- как это связано с другими частями проекта.

## Ограничения

Документ не должен:

- просто копировать код без объяснений;
- перескакивать между несвязанными файлами;
- использовать слишком краткие формулировки;
- предполагать, что читатель уже понимает Flutter-архитектуру.

## Критерии успеха

Документация считается успешной, если:

- новичок может читать код проекта в том же порядке, в котором работает приложение;
- по каждому важному файлу понятно, зачем он нужен;
- по каждому методу понятно, что он делает;
- по каждому экрану понятно, откуда в него приходит управление и куда оно уходит дальше;
- по моделям и репозиториям понятно, как проходят данные через приложение;
- весь проект воспринимается как единая связанная система, а не как набор разрозненных файлов.
